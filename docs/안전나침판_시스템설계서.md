## 1. 서비스 개요

### 1.1 한 줄 정의
**안전나침판**은 재난 발생 시 사용자의 현재 위치를 기준으로 **재난 심각도를 즉시 판단**하고, **알기 쉬운 행동요령**과 **가까운 대피소·병원·약국**을 안내하는 모바일 우선(mobile-first) 웹앱입니다.

### 1.2 핵심 가치
| 가치 | 설명 |
|---|---|
| 즉시성 | 앱 실행 즉시(3초 이내) 내 위치 기준 재난 단계와 날씨를 확인 |
| 이해 용이성 | 관공서 문체가 아닌, LLM이 쉬운 말로 바꾼 행동요령 제공 |
| 실행 가능성 | "알려주고 끝"이 아니라 대피소·병원·약국까지 길찾기로 연결 |
| 접근성 | 고령자·장애인도 쓸 수 있도록 음성 대화 지원 |

### 1.3 핵심 사용자 시나리오
1. **평상시**: 앱을 열면 오늘 날씨와 현재 지역의 재난 단계(대부분 "정상")를 확인 → 안심
2. **재난문자 수신 시**: 문자 속 링크를 눌러 앱 진입 → 문자에 담긴 위치가 자동 반영된 화면에서 현재 단계·행동요령을 바로 확인
3. **재난 중**: "지금 나가도 되나요?" 같은 질문을 대화창(텍스트/음성)에 입력 → 상황에 맞는 답변
4. **대피 필요 시**: 안전지도에서 가장 가까운 대피소를 확인하고 카카오맵 길찾기로 바로 이동

### 1.4 데이터/AI 소싱 원칙
- **재난문자**: 재난안전데이터포털 API
- **기상특보/동네예보**: 기상청 API 허브
- **대피소/시설**: 재난안전데이터포털 통합대피소 API
- **지도/길찾기/병원·약국 검색**: 카카오맵 API
- **재난 심각도 판정, 요약, 분류, 대화, 음성**: bizrouter를 통한 LLM(OpenAI 모델) 호출
- **행동요령 마스터 데이터**: 국민안전24 자연재난 24종 + 사회재난 33종 행동요령(기존 작업물 `재난유형별_국민행동요령.json`)과 단계구분 기준(`재난유형별_단계구분_기준.md`)을 Supabase에 적재하여 기준 데이터로 사용

## 2. 전체 시스템 아키텍처

```
[모바일 웹앱 클라이언트] (PWA, React 기반 권장)
        │  HTTPS / REST / Realtime(WebSocket)
        ▼
[Supabase 백엔드]
  ├─ Auth            (사용자/관리자 인증, Row Level Security 연동)
  ├─ Postgres DB      (+pgvector: 행동요령 임베딩 유사도 검색)
  ├─ Edge Functions   (외부 API 프록시, 스케줄러, LLM 호출 게이트웨이)
  ├─ Realtime         (재난 단계 변경 시 클라이언트에 실시간 push)
  └─ Storage          (원문 PDF/이미지, 관리자 업로드 자료)
        │
        ├──▶ 재난안전데이터포털 API  (긴급재난문자, 통합대피소)
        ├──▶ 기상청 API 허브         (동네예보, 기상특보)
        ├──▶ 카카오맵 API            (지도, 장소검색, 길찾기)
        └──▶ bizrouter (OpenAI 모델) (분류/요약/챗봇/STT·TTS)
```

### 2.1 계층별 역할

| 계층 | 역할 | 비고 |
|---|---|---|
| 클라이언트 | 화면 렌더링, 위치 획득(GPS/수동 선택), 카카오맵 SDK 렌더링, 음성 입출력 UI | React + Tailwind, PWA(오프라인 캐시로 최소 행동요령은 네트워크 없이도 열람 가능해야 함) |
| Supabase Edge Functions | 외부 API 키를 서버에서만 보관(클라이언트에 절대 노출 금지), 응답 정규화, LLM 호출 오케스트레이션, 배치/스케줄 작업(cron) | Deno 런타임 |
| Supabase DB | 원천 데이터 캐시, 판정 결과, 행동요령 마스터, 대화 로그, 사용자/관리자 | RLS로 사용자별 접근 제어 |
| 외부 API | 재난안전데이터포털, 기상청 API 허브, 카카오맵, bizrouter | 클라이언트는 이 API들을 직접 호출하지 않고 반드시 Edge Function을 경유 |

### 2.2 왜 클라이언트가 외부 API를 직접 호출하지 않는가
- 모든 외부 API 키(재난안전데이터포털 인증키, 기상청 API 허브 키, 카카오 REST API 키 중 서버용 키, bizrouter 키)는 **Edge Function 환경변수**로만 보관합니다.
- 카카오맵 **JavaScript SDK 키**(지도 렌더링용)만 예외적으로 클라이언트에 노출되며, 카카오 콘솔에서 **도메인 제한**을 반드시 설정합니다.
- 이렇게 하면 요청 빈도 제어(rate limit), 응답 캐싱, 응답 포맷 정규화를 서버 단에서 일괄 처리할 수 있습니다.

## 3. 정보구조(IA)와 하단 내비게이션

```
안전나침판
├─ 홈                (날씨 + 위치 + 재난상황 요약 + 빠른메뉴)
├─ 안전지도           (지도 기반 재난/대피소/병원/약국 통합 뷰)
├─ 공식 알림          (재난문자·기상특보 최신순 리스트)
├─ 행동요령           (LLM 분류 유형 → 상세 행동요령 → 대화 검색)
└─ (별도 진입) 관리자  (하단 탭에 노출하지 않음, /admin 경로 + 인증)
```

하단 탭 4개는 항상 고정 노출(sticky bottom nav), 관리자 메뉴는 일반 사용자 플로우에서 보이지 않고 별도 URL(`/admin`)과 로그인으로만 접근합니다.

## 4. 화면별 상세 설계

### 4.1 홈 화면 (`/`)

**목적**: 3초 내에 "지금 내가 있는 곳이 안전한지"를 파악시킨다.

**구성 (위에서 아래 순서, 모바일 세로 스크롤)**
1. **상단 위치 바**: `📍 세종특별자치시 어진동 ▾` — 탭하면 위치 선택 바텀시트 오픈(하단 4.6 참조)
2. **날씨 카드**: 기상청 동네예보 기반 — 현재기온/체감온도, 하늘상태 아이콘, 강수확률, 미세먼지, 일 최고/최저
3. **재난상황 카드**(가장 중요, 화면 상단 근처 고정감 있게 배치):
   - 배경색으로 5단계 표시(정상=회색/초록, 관심=파랑, 주의=노랑, 경계=주황, 심각=빨강)
   - 단계명 + 한 줄 요약(LLM 생성, 예: "인근 지역 호우경보 발효 중 — 외출을 자제하세요")
   - 탭하면 재난상황 상세 화면으로 이동(근거 원문, 관련 행동요령 링크)
   - **정상 단계일 때도 카드는 표시**하되 톤을 낮춰("현재 특별한 재난 상황이 없습니다") 사용자를 안심시킴
4. **빠른 메뉴**(아이콘 4개): 행동요령 / 대피소 찾기 / 병원·약국 / 공식 알림
5. **주변 안전시설 미리보기**: 가장 가까운 대피소 1곳 카드(거리, 도보시간) — 탭 시 안전지도로 이동

**데이터 소스**: 날씨(기상청 동네예보 캐시), 재난상황(disaster_situations 테이블, Realtime 구독), 대피소(shelters 테이블 + PostGIS 거리 계산)

**모바일 최적화 포인트**
- 뷰포트 375~430px 기준 1컬럼 레이아웃
- 재난상황 카드는 새 데이터 수신 시 subtle pulse 애니메이션(사용자가 변화를 인지하도록)
- 최초 진입 시 위치 권한 요청 → 거부 시 "시도/시군구 선택"으로 자연스럽게 유도(빈 화면 방지)

### 4.2 안전지도 (`/map`)

**목적**: 지도 위에서 재난 상황·대피소·병원·약국을 한 번에 겹쳐보고 길찾기로 연결.

**구성**
- 상단: 위치 표시/선택 바(홈과 동일 컴포넌트 재사용)
- 지도 영역(카카오맵 SDK, 화면의 65% 이상 차지)
- 지도 우상단 **레이어 토글 칩**: `재난상황` `대피소` `병원` `약국` (다중 선택 가능, 기본값: 재난상황 ON + 대피소 ON)
- 지도를 이동/확대하면 **"이 지역 재검색"** 버튼 노출(현재 화면 범위 기준으로 재조회 — 요구사항의 "화면에 보이는 지역에 표시") 
- 마커 클릭 → 하단 바텀시트: 시설명, 주소, 거리, 운영시간(병원/약국), **[길찾기]** 버튼(카카오맵 길찾기 딥링크로 이동)
- 재난상황은 마커가 아니라 **반투명 폴리곤/원형 오버레이**로 표시(시군구 단위 색상 = 5단계 색상)

**인터랙션 플로우**
```
위치 확정 → 현재 지도 bbox(경계 좌표) 산출
  → Edge Function 호출: 재난상황(해당 시군구), 대피소/병원/약국(bbox 내)
  → 마커/오버레이 렌더 → 사용자가 지도 이동 시 "이 지역 재검색" 노출
  → 마커 탭 → 상세 바텀시트 → [길찾기] → 카카오맵 앱/웹으로 이동
```

### 4.3 공식 알림 (`/alerts`)

**목적**: 원문 그대로의 재난문자·기상특보를 신뢰할 수 있는 타임라인으로 제공(LLM 가공 없이 원문 우선, 필요시 요약 토글).

**구성**
- 상단 필터: `전체` `긴급재난문자` `재난문자(속보)` `기상특보` + 지역 필터(현재 위치 기준 기본 ON)
- 리스트(최신순, 무한스크롤): 각 항목 = 유형 배지 + 발표시각 + 지역 + 제목/요약 1줄
- 항목 탭 → 상세: 원문 전체, (선택) "쉬운 설명 보기" 토글 시 LLM 요약 표시, 관련 행동요령 바로가기 버튼

### 4.4 행동요령 (`/guide`)

**목적**: "지금 내 상황에 맞는" 행동요령을 가장 먼저 보여주고, 그 외 유형은 탐색/검색으로 접근.

**구성**
1. 상단: **LLM이 분류한 현재 재난 유형 카드**(재난상황이 있을 때만 노출) — "현재 상황: 호우 · 유사 행동요령: 호우, 태풍" 형태로 유사도 높은 순 2~3개 유형 제시
2. **대화창(챗봇)**: 화면 중간 고정 — 텍스트 입력 + 마이크 버튼(음성 입력) — "지하주차장에 차를 대도 되나요?" 같은 자유질문 지원
3. 하단: 전체 57종 카테고리 목록(자연재난/사회재난 탭 구분, 검색창)을 아코디언 또는 그리드로 제공 — 사용자가 직접 유형을 선택해 상세 행동요령을 볼 수 있음
4. 상세 행동요령 화면: 기존 마스터 데이터(사전준비/발생시/이후 섹션)를 **LLM이 쉬운 말로 재작성한 버전**을 기본 표시하고, "원문 보기" 토글로 원본(safekorea 기반 문구)도 확인 가능

**대화창 상세 설계**는 6장에서 별도 기술합니다.

### 4.5 재난상황 상세 (공통 컴포넌트, 모달/전체화면)

홈·공식알림·안전지도 등 여러 화면에서 진입 가능한 공용 화면입니다.
- 현재 단계(1~5) 배지, 판정 근거 요약(LLM 생성 2~3문장)
- 판정에 사용된 원문 리스트(재난문자/기상특보) — 각각 펼쳐서 원문 확인 가능
- "이 상황에 맞는 행동요령 보기" CTA → 행동요령 상세로 이동
- "가까운 대피소 보기" CTA → 안전지도로 이동(대피소 레이어 활성 상태로 진입)

### 4.6 위치 선택 바텀시트 (공통 컴포넌트)

세 가지 선택 방법을 하나의 바텀시트에서 제공합니다.
1. **현재 위치 사용**(GPS) — 기본값, 브라우저 Geolocation API
2. **지도에서 선택** — 미니 카카오맵을 띄우고 사용자가 지도를 드래그해 중심점을 지정(리버스 지오코딩으로 행정동 표시)
3. **시도/시군구 직접 선택** — 2단 드롭다운(시/도 → 시/군/구), `regions` 테이블 기반

위치가 확정되면 `region_code`(법정동/행정동 코드)와 `lat/lng`를 전역 상태(Zustand/Context)에 저장하고, 이후 모든 API 호출(날씨, 재난상황, 대피소 검색 등)에 사용합니다.

## 5. 핵심 기능 — 재난상황 단계 판정 엔진

### 5.1 목표
"긴급재난문자, 재난문자(속보), 기상특보"라는 서로 다른 형식의 원천 데이터를 받아 **정상-관심-주의-경계-심각(1~5단계)** 로 정규화하고, 사용자가 이해할 수 있는 한 줄 요약을 만든다.

### 5.2 데이터 파이프라인

```
[스케줄러: Edge Function Cron, 3~5분 주기]
  1) 재난안전데이터포털 긴급재난문자 API 폴링
  2) 기상청 API 허브 기상특보 API 폴링
       ↓ (원문 저장)
  disaster_messages, weather_alerts 테이블 upsert
       ↓ (신규/변경분만 필터링)
  [판정 로직 실행] ── 규칙 기반 1차 판정 ──▶ [LLM 2차 판정/요약 (bizrouter)]
       ↓
  disaster_situations 테이블 upsert (지역코드별 최신 단계)
       ↓
  Supabase Realtime → 클라이언트 즉시 반영
```

### 5.3 단계 판정 로직 (하이브리드: 규칙 기반 + LLM)

**1차: 규칙 기반 매핑(결정적, 빠름, 우선 적용)**
사용자가 명시한 규칙을 그대로 코드화합니다.

| 원천 | 조건 | 매핑 단계 |
|---|---|---|
| 기상특보 | 주의보(강풍·호우·대설·풍랑·건조·폭풍해일·한파·태풍·황사) | 4단계(경계) |
| 기상특보 | 경보 | 5단계(심각) |
| 기상특보 | 폭염 중대경보 | 5단계(심각) |
| 긴급재난문자 | 발령유형이 "위급재난문자"(예: 대피명령, 공습경보 등) | 5단계(심각) |
| 긴급재난문자 | 발령유형이 "안전안내문자" 중 즉시대피/통제 키워드 포함 | 5단계(심각) |
| 재난문자 없음 + 특보 없음 | — | 1단계(정상) |

이 표는 앞서 정리한 `재난유형별_단계구분_기준.md`의 **기상특보 주의보/경보 기준**을 그대로 재사용합니다(강풍·호우·대설·풍랑·건조·폭풍해일·한파·태풍·황사).

**2차: LLM 판정(규칙으로 커버되지 않는 사회재난·복합상황 처리)**
- 대상: 재난문자 중 기상 특보로 환원되지 않는 것(화재, 시설사고, 감염병, 정전, 유해물질 유출 등) 및 여러 원문이 동시에 존재하는 복합 상황
- **입력**: 최근 N시간 이내 해당 지역의 재난문자 원문 목록 + 기상특보 목록(이미 1차 규칙 결과 포함)
- **출력 스키마(JSON, function calling/structured output 사용)**:
```json
{
  "level": 1,
  "level_name": "정상|관심|주의|경계|심각",
  "disaster_types": ["호우", "산사태"],
  "summary": "사용자에게 보여줄 2~3문장 요약(쉬운 말)",
  "reasoning": "판정 근거(내부 로깅용, 사용자 비노출 가능)",
  "confidence": 0.0
}
```
- **시스템 프롬프트 설계 원칙**:
  - "재난유형별_단계구분_기준.md"의 표(특히 위기경보 관심/주의/경계/심각 정의: *"위기징후 활동 수준과 확대 가능성"*)를 컨텍스트로 주입
  - 규칙 기반 1차 결과가 있으면 **그 값보다 낮은 단계로 내리지 않도록**(under-estimation 방지) 제약 — LLM은 상향 조정만 가능, 임의 하향은 근거가 명확할 때만 허용하고 관리자 검수 큐에 표시
  - 여러 재난유형이 동시 감지되면 **가장 높은 단계**를 최종 단계로 채택

### 5.4 지역 단위와 갱신 주기
- 판정은 **시군구 단위**(region_code)로 저장하며, 사용자의 현재 위치가 속한 region_code로 조회합니다.
- 갱신 주기: 평상시 5분, 활성 재난(4~5단계) 지역은 2분으로 단축(폴링 주기 자체를 상황에 맞게 조정하는 적응형 폴링).
- LLM 호출 비용 절감을 위해 **원문 해시값이 동일하면 재판정하지 않고 캐시된 결과 재사용**.

### 5.5 5단계 정의 요약(재사용)

| 단계 | 색상 | 정의 |
|---|---|---|
| 1. 정상 | 회색/초록 | 발효 중인 재난문자·특보 없음 |
| 2. 관심 | 파랑 | 위기징후 존재하나 활동 낮음(예비특보, 관심 단계 위기경보) |
| 3. 주의 | 노랑 | 위기징후 활동 증가(주의보 미만이나 사회재난 위기경보 "주의" 등) |
| 4. 경계 | 주황 | 기상 주의보 발효 또는 위기경보 "경계" 수준 |
| 5. 심각 | 빨강 | 기상 경보/중대경보 발효, 위급재난문자, 위기경보 "심각" |

## 6. 핵심 기능 — 행동요령 추천/대화 엔진

### 6.1 마스터 데이터
기존에 구축한 **자연재난 24종 + 사회재난 33종**(`재난유형별_국민행동요령.json`)을 `disaster_types` / `disaster_type_sections` 테이블로 적재합니다(10장 스키마 참고). 이 데이터가 모든 추천/검색의 근거(ground truth)입니다.

### 6.2 유사 유형 추천 (임베딩 기반)
1. 관리자가 마스터 데이터를 적재/수정할 때, 각 재난유형의 (이름+정의+행동요령 요약)을 bizrouter 임베딩 API로 벡터화하여 `disaster_types.embedding`(pgvector)에 저장
2. 5장에서 LLM이 분류한 `disaster_types`(예: ["호우","산사태"])가 마스터 데이터의 정확한 명칭과 다를 수 있으므로(LLM이 만든 자유 텍스트), **코사인 유사도 검색**으로 마스터 데이터 중 가장 가까운 유형 top-3를 매칭
3. 매칭 결과를 행동요령 화면 상단 카드에 노출(유사도 높은 순)

### 6.3 쉬운 말 변환/요약
- 마스터 데이터의 원문(관공서 문체)을 LLM으로 다음 원칙에 따라 재작성해 `disaster_type_sections.easy_text` 컬럼에 캐싱:
  - 한자어·전문용어 최소화, 문장은 15자 내외로 짧게
  - "~하여야 합니다" → "~하세요" 같은 명령형 단문으로 변환
  - 원문의 의미·수치(예: 24시간 신적설 20cm)는 절대 누락·왜곡하지 않음
- 재작성은 **관리자가 미리 배치로 생성**해두고(실시간 생성 아님), 관리자 콘솔에서 검수 후 게시하는 방식을 권장(안전 정보이므로 실시간 LLM 생성물을 검수 없이 바로 노출하지 않음)

### 6.4 대화형 검색(챗봇) — RAG 설계
```
사용자 질문(텍스트/음성)
   → (음성이면) STT 변환 (bizrouter)
   → 질문 임베딩 → disaster_type_sections에서 유사 문서 top-5 검색(pgvector)
   → 검색된 문서 + 사용자의 현재 재난상황(5장 결과) + 위치를 컨텍스트로 LLM 호출
   → 답변 생성(반드시 마스터 데이터 범위 내에서 답하도록 시스템 프롬프트에 명시,
      DB에 없는 내용은 "정확한 정보를 찾지 못했습니다. 119/재난안전상황실(044-205-1541~3)에 문의해 주세요"로 안내)
   → (음성 응답 요청 시) TTS 변환 (bizrouter) → 오디오 재생
```
- 모든 대화는 `chat_sessions`/`chat_messages`에 로깅(관리자가 오답·개선 필요 질의를 모니터링할 수 있도록)
- **답변 신뢰성이 중요한 도메인**이므로 RAG 근거 문서가 전혀 없을 경우 LLM이 임의로 지어내지 않고 위 안내 문구로 대체하도록 강하게 제약(그라운딩 실패 시 폴백)

### 6.5 음성 인터페이스
- 입력: 마이크 버튼 탭 → 브라우저 MediaRecorder로 녹음 → Edge Function으로 전송 → bizrouter STT
- 출력: 답변 텍스트를 bizrouter TTS로 변환(선택적, 사용자가 "읽어주기" 토글 시에만 — 데이터/비용 절감)
- 고령자 접근성을 고려해 **글자 크기 확대, 고대비 모드** 옵션을 설정에 함께 제공 권장

## 7. 핵심 기능 — 안전지도 & 안전시설

### 7.1 대피소
- 재난안전데이터포털 **통합대피소 API**를 주기적으로(예: 1일 1회, 데이터 갱신 빈도가 낮음) 동기화하여 `shelters` 테이블에 캐싱
- 대피소 유형(민방위대피소/지진옥외대피장소/이재민 임시주거시설/무더위쉼터/한파쉼터 등)을 `shelter_type` 컬럼으로 구분하고, **현재 재난 유형에 맞는 대피소 유형을 우선 표시**(예: 폭염 상황이면 무더위쉼터 우선)

### 7.2 병원/약국
- 카카오맵 **키워드/카테고리 장소검색 API**(category_group_code: `HP8`=병원, `PM9`=약국)를 실시간 호출(현재 위치 or 선택 위치 기준 반경 검색)
- 대피소와 달리 실시간성이 중요하지 않으므로 짧은 캐시(예: 10분)만 적용해 API 호출량 절감
- 응급의료 정보(야간/공휴일 운영 여부)는 필요시 **응급의료포털 API**(공공데이터포털) 연동을 확장 검토(로드맵 참고)

### 7.3 길찾기 연동
- 마커 상세의 [길찾기] 버튼은 카카오맵의 **길찾기 웹/앱 딥링크**(`https://map.kakao.com/link/to/{name},{lat},{lng}`)로 연결 — 별도 자체 경로 계산 로직 불필요, 카카오맵 앱이 설치되어 있으면 앱으로 전환

### 7.4 지도 클러스터링/성능
- 마커가 많은 도심 지역은 카카오맵 SDK의 클러스터러(Clusterer) 사용
- 재난상황 오버레이는 시군구 폴리곤 GeoJSON을 서버에서 미리 단순화(simplify)해 전송(모바일 렌더링 성능 고려)

## 8. 핵심 기능 — 공식 알림

- `disaster_messages`, `weather_alerts` 원문을 그대로 페이지네이션 조회(최신순 `issued_at DESC`)
- 지역 필터는 재난문자의 `region_codes`(배열) 또는 특보의 `region_codes`에 사용자 region_code가 포함되는지로 필터링
- "쉬운 설명 보기"는 6.3의 LLM 요약과 다른 용도(개별 원문 1건 요약)이므로 온디맨드로 생성 후 `easy_summary` 컬럼에 캐싱(같은 원문 재요청 시 재생성하지 않음)

## 9. 핵심 기능 — 관리자 콘솔 (`/admin`)

### 9.1 인증
- Supabase Auth(이메일/비밀번호 또는 매직링크)로 로그인
- `profiles.role = 'admin'`인 사용자만 `/admin` 라우트 접근 허용(프론트 라우트 가드 + **Supabase RLS로 서버 단에서도 이중 검증**, 프론트 가드만 믿지 않음)

### 9.2 메뉴 구성
1. **대시보드**: 오늘 API 호출 현황, LLM 사용량/비용, 현재 활성 재난상황(4~5단계) 지역 수
2. **테이블 뷰어**: 주요 테이블(disaster_types, shelters, disaster_situations 등) 목록/검색/수정(CRUD) — Supabase의 자동 생성 REST(PostgREST)를 감싼 간단한 그리드 UI
3. **API 연동 상태**: 4개 외부 API(재난안전데이터포털, 기상청 API 허브, 카카오맵, bizrouter) 각각의 최근 호출 성공/실패 로그, 마지막 성공 시각, 남은 쿼터(가능한 경우)
4. **API 수동 테스트 도구**: 각 API에 대해 파라미터를 입력해 즉시 호출해보고 원본 응답(raw JSON)을 확인하는 폼 — 장애 대응/디버깅용
5. **LLM 판정 검수 큐**: 5.3의 "LLM이 규칙 기반보다 하향 조정한 케이스", "confidence 낮은 케이스"를 모아 관리자가 승인/수정
6. **행동요령 easy_text 검수**: 6.3에서 생성된 쉬운말 버전을 원문과 나란히 놓고 승인/수정/재생성
7. **대화 로그 모니터링**: 6.4 챗봇의 "근거 없음" 응답 비율, 최근 질문 목록(개선 포인트 파악용)

### 9.3 감사 로그
모든 관리자 CRUD/승인 행위는 `admin_audit_logs`에 기록(누가, 언제, 무엇을 변경했는지) — 안전 정보를 다루는 서비스이므로 필수.

## 10. 데이터베이스 설계 (Supabase/Postgres)

### 10.1 ERD 개요

```
regions ──┬── disaster_situations ──── disaster_messages
          │                       └─── weather_alerts
          ├── shelters
          └── (사용자 위치 조회 기준)

disaster_types ──< disaster_type_sections
      │  (embedding: pgvector)
      └──< disaster_type_levels  (5장의 단계구분 기준 근거 데이터)

profiles(=auth.users 확장) ──< chat_sessions ──< chat_messages
profiles ──< admin_audit_logs

api_call_logs (모든 외부 API 호출 공통 로그)
```

### 10.2 주요 테이블 정의 (DDL 예시)

```sql
-- 지역 코드 (행정안전부 법정동/행정동 코드 기준)
create table regions (
  region_code    text primary key,          -- 예: 5011000000 (세종시)
  sido           text not null,             -- 시도명
  sigungu        text,                      -- 시군구명 (세종시는 sigungu 없음 가능)
  lat            double precision,
  lng            double precision,
  kma_nx         int,                       -- 기상청 동네예보 격자 X
  kma_ny         int                        -- 기상청 동네예보 격자 Y
);

-- 긴급재난문자 원문 (재난안전데이터포털)
create table disaster_messages (
  id              uuid primary key default gen_random_uuid(),
  external_id     text unique,              -- 포털 제공 고유 ID (중복 수집 방지)
  msg_type        text,                     -- 긴급재난문자 / 안전안내문자 / 위급재난문자 등
  region_codes    text[] not null,
  content         text not null,
  issued_at       timestamptz not null,
  raw_payload     jsonb,                    -- 원본 응답 전체 보관
  created_at      timestamptz default now()
);
create index on disaster_messages using gin (region_codes);

-- 기상특보 원문 (기상청 API 허브)
create table weather_alerts (
  id              uuid primary key default gen_random_uuid(),
  external_id     text unique,
  alert_kind      text not null,            -- 강풍/호우/대설/한파/태풍/폭염/황사/건조/풍랑/폭풍해일 등
  alert_level     text not null,            -- 예비특보/주의보/경보/중대경보
  region_codes    text[] not null,
  content         text,
  issued_at       timestamptz not null,
  effective_until timestamptz,
  raw_payload     jsonb,
  created_at      timestamptz default now()
);
create index on weather_alerts using gin (region_codes);

-- LLM/규칙 하이브리드 판정 결과 (지역별 "현재" 재난상황)
create table disaster_situations (
  region_code     text primary key references regions(region_code),
  level           smallint not null check (level between 1 and 5), -- 1정상~5심각
  level_name      text not null,
  disaster_type_ids uuid[],                 -- 매칭된 disaster_types.id 배열

  confidence      numeric,
  source_message_ids uuid[],                -- 근거가 된 disaster_messages
  source_alert_ids   uuid[],                -- 근거가 된 weather_alerts
  needs_review    boolean default false,    -- 관리자 검수 필요 여부 (5.3 하향조정 등)
  updated_at      timestamptz default now()
);

-- 행동요령 마스터: 재난유형 (자연재난24 + 사회재난33)
create table disaster_types (
  id              uuid primary key default gen_random_uuid(),

  code            text unique not null,     -- 예: typhoon, earthquake, fire ...
  name_ko         text not null,
  source          text,
  embedding       vector(1536),             -- bizrouter 임베딩 모델 차원에 맞춤

create table disaster_type_sections (

  review_status   text default 'draft' check (review_status in ('draft','approved')),
  embedding       vector(1536),
  sort_order      int
);

-- 재난유형별 단계구분 기준 (이전 조사 결과 근거 데이터)
create table disaster_type_levels (

  level_label     text not null,            -- 주의보/경보 또는 관심/주의/경계/심각

-- 대피소 (통합대피소 API 캐시)
create table shelters (

  name            text not null,

  address         text,

create table chat_sessions (

-- 관리자 감사 로그
create table admin_audit_logs (
  id              uuid primary key default gen_random_uuid(),
```

### 10.3 RLS(Row Level Security) 정책 개요
| 테이블 | 일반 사용자 | 관리자 |
|---|---|---|
| disaster_messages / weather_alerts / disaster_situations / shelters / disaster_types(및 하위) | **읽기만 허용**(select) | 전체 허용(insert/update/delete) |
| chat_sessions / chat_messages | 본인 세션만 읽기/쓰기 (`auth.uid() = user_id`) | 전체 조회(모니터링용) |
| api_call_logs / admin_audit_logs | 접근 불가 | 전체 허용 |
| profiles | 본인 행만 읽기/수정 | 전체 읽기, role 변경은 최고관리자만(추가 정책 필요) |

일반 사용자의 원천 데이터 쓰기는 전혀 없고(모두 서버/스케줄러/관리자가 기록), 이는 **Edge Function이 service_role 키로 우회**하므로 RLS 정책상 anon/authenticated 역할에는 읽기 권한만 부여하면 충분합니다.

## 12. 딥링크/URL 파라미터 설계 (재난문자 연동)

재난문자에 포함된 링크를 통해 앱에 진입할 때, 문자 자체의 위치 정보를 그대로 반영해야 합니다.

**URL 스펙 예시**
```
https://safety-compass.app/?region=5011000000&lat=36.4800&lng=127.2890&src=cbs
```
- `region`: 법정동/행정동 코드 (가장 신뢰도 높은 값, 있으면 최우선 사용)
- `lat`/`lng`: 좌표(문자 시스템이 좌표만 제공하는 경우) — 있으면 리버스 지오코딩으로 region_code 역산
- `src`: 유입 출처 태깅(`cbs`=재난문자, `push`=자체 알림 등) — 분석용

**처리 로직**
1. 앱 진입 시 URL 쿼리 파라미터 우선 파싱
2. `region`이 있으면 즉시 해당 지역으로 위치 확정(GPS 요청 생략 가능, 단 사용자가 원하면 홈에서 위치 재선택 가능)
3. `region`이 없고 `lat/lng`만 있으면 카카오맵 리버스 지오코딩으로 region_code 조회
4. 확정된 위치로 홈 화면의 모든 데이터(날씨/재난상황/대피소)를 즉시 로드
5. 파라미터가 전혀 없으면 4.6의 일반 위치 선택 플로우로 진행

## 13. 비기능 요구사항

### 13.1 모바일 최적화
- 뷰포트 기준 360~430px 우선 설계, 태블릿/데스크톱은 중앙 정렬 단일 컬럼으로 자연 확장
- 하단 내비게이션은 iOS Safe Area(`env(safe-area-inset-bottom)`) 대응
- PWA manifest + Service Worker로 "홈 화면에 추가" 지원, 핵심 행동요령(easy_text)은 오프라인 캐시

### 13.2 성능
- 재난상황/대피소 등 원천 데이터는 서버에서 캐싱 후 제공(외부 API 직접 호출 지양)
- LLM 호출은 비용/지연 문제로 **가능한 규칙 기반으로 선(先)처리**, LLM은 규칙으로 커버 안 되는 부분만 최소 호출
- 지도 마커/폴리곤은 뷰포트 내 데이터만 로드(페이지네이션/클러스터링)

### 13.3 보안
- 모든 외부 API 키는 Edge Function 환경변수(서버사이드)로만 보관
- 관리자 라우트는 RLS + 프론트 라우트 가드 이중화
- 사용자 위치정보는 세션/로컬 상태로만 보관, 서버 저장 시에는 최소한(챗봇 세션의 region_code 정도)으로 제한하고 보관기간 정책 수립

### 13.4 접근성
- 음성 입출력(6.5), 고대비 모드, 큰 글씨 모드
- 색상만으로 단계를 구분하지 않고 **텍스트 라벨(정상/관심/주의/경계/심각)을 항상 함께 표기**(색약 사용자 고려)

### 13.5 신뢰성(재난 서비스 특성상 중요)
- LLM 생성 콘텐츠(easy_text, 챗봇 답변)는 원문 대비 사실 왜곡이 없는지 **관리자 검수 프로세스**를 반드시 거치도록(6.3, 9.2-6)
- 외부 API 장애 시 폴백: 마지막으로 성공한 캐시 데이터를 "OO분 전 기준" 라벨과 함께 표시(빈 화면 금지)

## 14. 개발 로드맵 (MVP → 확장)

**MVP (1차)**
- 홈(날씨+위치+재난상황 카드), 공식 알림, 행동요령(마스터 데이터 열람만, 대화 제외), 안전지도(대피소만)
- 재난상황 판정: 규칙 기반(기상특보 주의보/경보 매핑)만 우선 구현, LLM 판정은 2차로 확장
- 관리자: 테이블 뷰어 + API 상태 모니터링만

**2차 확장**
- LLM 하이브리드 판정(사회재난 포함), easy_text 변환 및 검수 워크플로우
- 대화형 검색(텍스트), 유사유형 추천(임베딩)
- 안전지도에 병원/약국 레이어 추가, 길찾기 연동

**3차 확장**
- 음성 입출력(STT/TTS)
- PWA 오프라인 지원, 푸시 알림(재난상황 등급 변경 시)
- 관리자 LLM 검수 큐 고도화, 사용 통계 대시보드
